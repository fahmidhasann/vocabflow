Three Implementation Approaches

All use the Oxbridge aesthetic. The difference is in how deeply we rework the layout and component architecture.

A

Token Swap

Replace CSS variables, font stack, and color palette globally. All existing components inherit the new look with minimal structural changes. Quickest path, lower risk.

Pros

  • Low risk — existing layouts unchanged
  • Fast to implement
  • Easy to revert

Cons

  • Misses opportunities for layout improvements
  • Card shapes and spacing stay the same
  • Typography hierarchy limited to what's already there
B

Component Refresh RECOMMENDED

New design tokens plus redesigned core components: cards, buttons, badges, the review card flip, and the bottom nav. Pages stay structurally the same but components get Oxbridge-native markup and spacing.

Pros

  • Components built for the aesthetic, not retrofitted
  • Typography hierarchy properly expressed
  • Serif headings, parchment textures, amber accents land correctly
  • Manageable scope — no routing or data changes

Cons

  • More files touched than Option A
  • Review card 3D flip needs rethinking for the new style
C

Full Overhaul

Redesign tokens, components, and page-level layouts. Introduces new layout patterns: e.g. a split-pane word detail view, a magazine-grid word browser, a richer stats dashboard. Maximum creative expression.

Pros

  • Most distinctive and memorable result
  • Every page feels purpose-built
  • Room for editorial layout moments

Cons

  • Largest scope — more things to coordinate
  • Higher chance of layout regressions on mobile
  • Takes longer to implement correctly
